## 7.1 推理时间计算（Test-Time Compute）

OpenAI o1 系列和 DeepSeek-R1 开创了"推理时多花时间换更好结果"的范式：

$$\text{答案质量} \propto f(\text{推理时计算量})$$

这是 LLM 发展中最重要的范式转移之一——从"训练时 Scaling"到"推理时 Scaling"。

|------|------|------|------|------|
| **内部 CoT** | o1, R1 | 训练模型生成长思维链 | 数学/代码大幅提升 | 推理时 tokens 增加 5-50× |
| **外部搜索** | AlphaProof | MCTS + 验证器 | 竞赛数学 IMO 银牌 | 需要搜索树 + 验证器 |

## 7.2 MoE：混合专家的规模化之路

### 7.2.1 MoE 基本原理

MoE（Mixture of Experts）是扩大模型容量而不等比增加计算量的关键技术：

$$y = \sum_{i=1}^{N} g_i(x) \cdot E_i(x), \quad g(x) = \text{TopK}(\text{softmax}(W_g x))$$

### 7.2.2 MoE 架构演进

| 模型 | 总参数 | 激活参数 | 专家数 | Top-K | 路由方式 |
|------|--------|---------|--------|-------|---------|
| Switch Transformer | 1.6T | 同 T5-Base | 2048 | 1 | Token-level |
| GShard | 600B | ~3B | 64 | 2 | Token-level |
| Mixtral 8×7B | 47B | 13B | 8 | 2 | Token-level |
| DeepSeek-V3 | 671B | 37B | 257 (1 shared) | 8 | 无辅助损失 |
| DeepSeek-V3 | ~2T | ~50B | 2048 | 8-16 | 新型混合架构 |
| Llama 4 Behemoth | 288B | ~50B | 16 | 2 | Expert-choice |

### 7.2.3 DeepSeek-V3 的 MoE 创新

DeepSeek-V3 在 MoE 路由上有两个关键创新：

1. **无辅助损失负载均衡**：不使用额外损失函数来平衡专家负载，而是通过 bias 项动态调整
2. **细粒度专家**：256 个小专家比 16 个大专家效果更好

```python
class DeepSeekMoE(nn.Module):
    def __init__(self, dim, num_experts=256, num_shared=1, top_k=8):
        super().__init__()
        self.shared_experts = nn.ModuleList([Expert(dim) for _ in range(num_shared)])
        self.routed_experts = nn.ModuleList([Expert(dim) for _ in range(num_experts)])
        self.gate = nn.Linear(dim, num_experts)
        self.expert_bias = nn.Parameter(torch.zeros(num_experts))  # 动态负载均衡
    
    def forward(self, x):
        # 共享专家始终激活
        shared_out = sum(e(x) for e in self.shared_experts)
        # 路由专家 Top-K 选择
        scores = self.gate(x) + self.expert_bias  # bias 辅助负载均衡
        top_k_scores, top_k_indices = scores.topk(self.top_k)
        weights = F.softmax(top_k_scores, dim=-1)
        routed_out = sum(w * self.routed_experts[i](x) 
                        for w, i in zip(weights.unbind(-1), top_k_indices.unbind(-1)))
        return shared_out + routed_out
```

训练成本仅 $5.57M（2048 × H800，60 天），是同等性能密集模型的 **1/10**。

### 7.3.2 长上下文的三大挑战

1. **注意力复杂度**：$O(n^2)$ 在 1M+ token 时不可承受，128K 需要 ~1.6TB 注意力矩阵
2. **"迷失在中间"**（Lost in the Middle）：模型对长文本中间部分的关注度下降
3. **KV Cache 显存**：128K 上下文的 KV Cache 可能超过模型本身

### 7.3.3 解决方案全景

**位置编码外推**：

```python
# RoPE 位置编码
def rotary_embedding(x, seq_len, base=10000):
    dim = x.shape[-1]
    freqs = 1.0 / (base ** (torch.arange(0, dim, 2) / dim))
    t = torch.arange(seq_len)
    freqs = torch.outer(t, freqs)
    return torch.polar(torch.ones_like(freqs), freqs)

# YaRN 外推：调整频率基数实现长度外推
def yarn_rope(x, seq_len, base=10000, scale=4):
    """从 4K 外推到 128K（scale=32）"""
    adjusted_base = base * (scale ** (dim / (dim - 2)))
    return rotary_embedding(x, seq_len, base=adjusted_base)
```

**Ring Attention**：

```
GPU 0: [Chunk 1] → Attend(Q1, K1V1) → Pass K1V1 to GPU 1
GPU 1: [Chunk 2] → Attend(Q2, K2V2) + Attend(Q2, K1V1) → Pass K2V2 to GPU 2
GPU 2: [Chunk 3] → Attend(Q3, K3V3) + Attend(Q3, K1K2V1V2) → ...
```

**稀疏注意力**：

- **滑动窗口**：只关注最近 $w$ 个 token + 全局 anchor token
- **H2O (Heavy Hitter Oracle)**：保留注意力分数最高的 token，丢弃不重要的
- **SnapKV**：只保留每层注意力最集中的 KV 对

## 7.4 高效架构：Mamba 与混合模型

### 7.4.1 Mamba（状态空间模型）

Mamba 基于结构化状态空间模型（S4），实现**线性复杂度** $O(n)$ 的序列建模：

$$h_t = \bar{A} h_{t-1} + \bar{B} x_t, \quad y_t = C h_t + D x_t$$

关键创新：**选择性机制（Selective Mechanism）**——让 $\bar{B}$、$C$、$\Delta$ 依赖于输入，使模型能选择性地记忆或遗忘。

```python
# Mamba 选择性 SSM 核心（简化）
class SelectiveSSM(nn.Module):
    def __init__(self, d_model, d_state=16):
        self.A = nn.Parameter(torch.randn(d_model, d_state))
        self.D = nn.Parameter(torch.ones(d_model))
        # B, C, dt 由输入动态生成（选择性的关键）
        self.x_proj = nn.Linear(d_model, d_state * 2 + 1)
    
    def forward(self, x):  # x: [B, L, D]
        B, C, dt = self.x_proj(x).split([d_state, d_state, 1], dim=-1)
        dt = F.softplus(dt)  # 离散化步长
        A_bar = torch.exp(self.A * dt)  # 离散化
        B_bar = B * dt
        # 递推计算（实际用并行扫描算法加速）
        h = torch.zeros_like(x[:, 0])
        outputs = []
        for t in range(x.shape[1]):
            h = A_bar * h + B_bar[:, t] * x[:, t]
            y = (C[:, t] * h).sum(-1) + self.D * x[:, t]
            outputs.append(y)
        return torch.stack(outputs, dim=1)
```

$$\text{第 } n \text{ 代模型} = f(\text{第 } n-1 \text{ 代合成数据}) \rightarrow \text{分布塌缩}$$

## 7.6 无 Tokenizer 架构：BLT

### 7.6.1 为什么要去掉 Tokenizer？

传统 Tokenizer（BPE/SentencePiece）存在根本性缺陷：

- **语言偏见**：英文 1 token ≈ 4 chars，中文/日文 1 token ≈ 1-2 chars
- **不可逆压缩**：某些字符组合无法正确拆分
- **词汇外问题**：新词、代码、数学符号可能被错误分词

### 7.6.2 Byte Latent Transformer (BLT)

Meta FAIR 的 BLT (2024) 提出无 Tokenizer 的字节级 Transformer：

```
传统 Pipeline:
  文本 → Tokenizer → Token IDs → Transformer → Token IDs → Detokenizer → 文本

BLT Pipeline:
  文本 → UTF-8 字节 → 动态分组 Patch → Latent Transformer → Patch → 字节 → 文本
```

关键创新是**动态 Patch 分组**——根据字节的熵值自动决定分组边界：

$$\text{Patch boundary at byte } b_t \text{ if } H(b_{t+1} | b_{1:t}) > \tau$$

高熵位置（不可预测的地方）作为分组边界，低熵连续字节合并为一个 patch。

在同等计算量下，BLT 超越了 Llama 3 的 BPE tokenizer 方案。

class SparseAutoencoder(nn.Module):
    """提取模型内部的可解释特征"""
    def __init__(self, d_model, d_dict):
        self.encoder = nn.Linear(d_model, d_dict)
        self.decoder = nn.Linear(d_dict, d_model)
    
    def forward(self, x):
        # 稀疏编码：大部分特征为 0
