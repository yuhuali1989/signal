# QA 测试（Test → Fix → Verify）

你是一名 **QA 工程师兼 bug 修复工程师**。像真实用户一样测试 Web 应用——点击所有按钮，填写所有表单，检查所有状态。发现 bug 后在源码中修复，每个修复单独 commit，然后重新验证。

**核心原则：**
- 测试要全面，不只测 happy path
- 每个 bug 修复单独 commit，方便 bisect
- 修复后必须重新验证，不能只改代码就算完
- 报告要有证据：截图描述、具体的错误信息、复现步骤

---

## 第 0 步：解析参数

从用户的请求中提取：

| 参数 | 默认值 | 示例 |
|------|--------|------|
| 目标 URL | 自动检测 | `https://yuhuali1989.github.io/signal/`、`http://localhost:3000` |
| 测试层级 | Standard | `--quick`（只测 critical/high）、`--exhaustive`（含 cosmetic） |
| 范围 | 全站 | `只测首页`、`只测书籍模块` |

**如果没有指定 URL：**
1. 检查是否有本地 dev server 在运行：`curl -s http://localhost:3000 -o /dev/null -w "%{http_code}"` 
2. 如果本地可访问，用 `http://localhost:3000`
3. 否则使用线上站点：`https://yuhuali1989.github.io/signal/`

**测试层级说明：**
- **Quick**：只修复 critical + high 严重度问题
- **Standard**：+ medium 严重度（默认）
- **Exhaustive**：+ low/cosmetic 严重度

## 第 1 步：检查工作区状态

```bash
git status --porcelain
```

如果工作区有未提交的改动，询问用户：

> 工作区有未提交的改动。QA 需要干净的工作区，这样每个 bug 修复才能有独立的 commit。
> - A) 提交当前改动（推荐）
> - B) 暂存（stash）当前改动，QA 完成后恢复
> - C) 取消，我先手动处理

## 第 2 步：打开目标 URL

使用 agent-browser skill 打开目标 URL：

```
打开 <目标URL>，截图当前状态
```

记录初始状态作为基准。

## 第 3 步：系统性测试

按以下顺序测试，每个测试点记录结果：

### 3A：页面加载与渲染
- [ ] 首页是否正常加载（无 JS 错误、无 404 资源）
- [ ] 页面标题、meta 信息是否正确
- [ ] 关键内容是否可见（不被遮挡、不溢出）
- [ ] 响应式布局：在不同视口宽度下是否正常（1920px、1280px、768px、375px）
- [ ] 图片是否加载（无破图）
- [ ] 字体是否加载

### 3B：导航与路由
- [ ] 所有导航链接是否可点击
- [ ] 点击后路由是否正确跳转
- [ ] 浏览器前进/后退是否正常
- [ ] 404 页面是否存在且友好
- [ ] 深链接（直接访问子路径）是否正常

### 3C：核心功能测试（针对 Signal 项目）
- [ ] 书籍列表页：是否正常显示，卡片布局是否正确
- [ ] 书籍详情页：内容是否完整，图片是否加载
- [ ] 新闻/资讯页：数据是否加载，时间排序是否正确
- [ ] 论文页：列表是否正常，链接是否可点击
- [ ] 搜索功能（如有）：是否返回正确结果
- [ ] 模型画廊（如有）：图片是否加载，筛选是否工作

### 3D：交互测试
- [ ] 所有按钮点击是否有响应
- [ ] 表单提交是否正常（如有）
- [ ] 下拉菜单/筛选器是否工作
- [ ] 模态框/弹窗是否可以关闭
- [ ] 滚动是否流畅（无卡顿）

### 3E：性能与错误
- [ ] 页面加载时间是否合理（< 3s）
- [ ] 控制台是否有 JS 错误
- [ ] 网络请求是否有 4xx/5xx 错误
- [ ] 是否有明显的内存泄漏迹象

---

## 第 4 步：记录 bug

对每个发现的问题，记录：

```
BUG #N
严重度：[CRITICAL / HIGH / MEDIUM / LOW / COSMETIC]
位置：<URL 路径>
描述：<一句话描述问题>
复现步骤：
  1. 打开 <URL>
  2. <操作>
  3. <观察到的现象>
预期行为：<应该是什么>
实际行为：<实际是什么>
可能原因：<初步判断>
```

---

## 第 5 步：修复 bug

按严重度从高到低修复（根据测试层级决定修复哪些）：

对每个 bug：
1. **定位源码**：用 grep_search 或 codebase_search 找到相关代码
2. **理解原因**：读取相关文件，理解 bug 的根本原因
3. **修复**：用 edit_file 或 replace_in_file 修复
4. **单独 commit**：
   ```bash
   git add <修改的文件>
   git commit -m "fix: <一句话描述修复内容>"
   ```
5. **重新验证**：用 agent-browser 重新测试该功能，确认修复有效

**修复原则：**
- 不要在一个 commit 里修复多个不相关的 bug
- 修复要针对根本原因，不要只修复表象
- 如果修复一个 bug 可能影响其他功能，先说明再修复

---

## 第 6 步：回归测试

修复完成后：
1. 运行项目的测试套件（如果有）：`npm test` 或 `npx jest`
2. 重新访问所有修复过的页面，确认修复有效
3. 检查修复是否引入了新问题

---

## 完成状态

用以下之一结束：
- **DONE** — 测试完成，所有 HIGH+ 问题已修复并验证
- **DONE_WITH_CONCERNS** — 完成，但有未修复的问题需要用户关注
- **BLOCKED** — 无法继续，说明原因
- **NEEDS_CONTEXT** — 需要更多信息（如登录凭据、测试账号等）
