# 第 1 章：核心理念——当 AI 不该碰你的数据

> 选自《Code as Proxy — AI 时代的数据安全架构》

## 1.1 一个根本性矛盾

2026 年，AI 已经无处不在。Claude、GPT、Gemini 等大模型能写代码、做分析、生成报告，效率提升 3-5 倍已是常态。但一个根本性矛盾正在撕裂每一个企业的技术决策：

> **AI 越强大，它需要接触的数据越多；数据越敏感，你越不敢让 AI 碰它。**

这不是一个可以用"加密"或"脱敏"简单解决的问题。传统的 AI 助手工作模式是这样的：

```
用户数据 ──→ LLM（外部服务器）──→ 分析结果
         ↑                        ↑
    数据离开了你的边界        模型"看到"了你的数据
```

当你把一段核心算法代码粘贴到 Claude 里调试时，这段代码已经离开了你的控制范围。当你让 GPT-4 分析一份客户交易数据时，这些数据已经到了 OpenAI 的服务器上。

**这就是 AI 时代的"数据引力"困境**：数据有引力，它倾向于留在原地；AI 有引力，它倾向于吸收一切数据。两种引力的对抗，构成了当代企业技术架构的核心张力。

## 1.2 真实世界的代价

这不是理论推演。真实的代价已经在发生：

| 时间 | 事件 | 后果 |
|------|------|------|
| 2023-04 | Samsung 工程师将芯片源代码粘贴到 ChatGPT | 核心 IP 泄露，全面禁用 ChatGPT |
| 2024-01 | 某金融公司员工用 GPT-4 分析客户交易数据 | 违反 GDPR，被罚 €2M，CTO 辞职 |
| 2024-06 | OpenAI 安全事件，企业 API 调用日志短暂暴露 | 多家企业紧急审查 API 调用历史 |
| 2025-01 | 某公司专利算法发送给 ChatGPT | 专利新颖性被质疑，申请被驳回 |
| 2025-03 | 员工通过 Cursor 将整个代码仓库索引发送到外部 API | 紧急部署代码扫描网关 |

每一个案例背后，都是同一个模式：**AI 需要理解数据才能工作，但理解数据意味着数据必须离开安全边界。**

## 1.3 Code as Proxy：第三条路

面对"用 AI 但数据泄露"和"不用 AI 但效率落后"的两难，业界需要第三条路。

**Code as Proxy（代理代码执行）** 就是这第三条路。它的核心思想可以用一句话概括：

> **"Don't give AI your data. Give AI the ability to write code that touches your data."**
>
> 不要把数据给 AI，而是让 AI 写代码来操作你的数据。

这个看似简单的思路转换，实际上是一次范式级的架构变革：

```
传统模式：
  用户数据 ──→ LLM ──→ 结果
  （数据离开边界，LLM 直接接触原始数据）

Code as Proxy 模式：
  数据 Schema ──→ LLM ──→ 生成代码 ──→ 本地执行 ──→ 结果摘要
  （数据不出域，LLM 只看到结构描述，代码在本地执行）
```

**AI 理解的是数据的"形状"（Schema），而非数据的"内容"（Values）。** 它根据这个形状生成操作代码，代码被送回数据所在的环境执行，AI 最终只看到执行结果的摘要。

## 1.4 一个直观的类比

想象你是一个盲人侦探（AI），需要调查一个密室（数据库）里发生了什么。

**传统模式**：有人把密室里的所有物品搬到你面前，你逐一触摸检查。问题是，物品离开了密室，可能被其他人看到或拿走。

**Code as Proxy 模式**：有人给你一份密室的平面图和物品清单（Schema），你根据这些信息写出一份调查指令（Code）："检查桌子上的第三个抽屉，统计里面文件的数量和日期范围"。然后有人拿着你的指令进入密室执行，回来告诉你："抽屉里有 47 份文件，日期从 2024-01 到 2025-03"。

你从未进入密室，从未触碰任何物品，但你完成了调查。

## 1.5 Code as Proxy 的四层架构

Code as Proxy 不是一个单一技术，而是一套分层架构：

```
┌─────────────────────────────────────────────┐
│  第 1 层：业务语义层 (Ontology)              │
│  AI 理解的不是原始数据，而是业务对象          │
│  （订单/用户/设备）及其关系和约束             │
├─────────────────────────────────────────────┤
│  第 2 层：代理生成层 (Code / Action Gen)     │
│  AI 根据自然语言意图，生成操作 Ontology       │
│  的代码或结构化 Action                       │
├─────────────────────────────────────────────┤
│  第 3 层：受控执行层 (Sandboxed Runtime)     │
│  代码在安全沙箱中执行，权限受 ACL 约束        │
│  数据不出域，执行环境隔离                     │
├─────────────────────────────────────────────┤
│  第 4 层：结果摘要层 (Result Summary)        │
│  AI 只收到执行结果的摘要（聚合统计/状态变更）  │
│  不接触原始行数据                            │
└─────────────────────────────────────────────┘
```

每一层都有明确的安全边界：

- **第 1 层**解决"AI 理解什么"——不是原始数据，而是业务语义
- **第 2 层**解决"AI 产出什么"——不是直接操作，而是代码/指令
- **第 3 层**解决"代码在哪执行"——不是 AI 侧，而是数据侧
- **第 4 层**解决"AI 看到什么"——不是原始结果，而是摘要

## 1.6 与传统安全方案的本质区别

Code as Proxy 不是对传统安全方案的改良，而是一种范式转换：

| 维度 | 传统方案 | Code as Proxy |
|------|---------|---------------|
| **数据流向** | 数据 → AI | Schema → AI → Code → 数据 |
| **AI 接触面** | 原始数据 | 仅 Schema + 结果摘要 |
| **安全假设** | "我信任 AI 不会泄露" | "AI 根本看不到数据" |
| **加密策略** | 传输加密 + 静态加密 | 数据不移动，无需传输加密 |
| **脱敏策略** | 发送前脱敏 | 无需脱敏（数据不发送） |
| **审计粒度** | 审计"谁访问了数据" | 审计"谁生成了什么代码" |
| **合规难度** | 需证明数据安全传输和存储 | 数据不出域，天然合规 |
| **幻觉风险** | AI 可能基于数据产生错误推断 | AI 操作有类型约束的对象，幻觉率大幅降低 |

关键区别在于**安全假设的根本不同**：

- 传统方案假设"AI 是可信的，但需要防护措施"
- Code as Proxy 假设"AI 是不可信的，所以不让它碰数据"

这是一种**零信任（Zero Trust）**思想在 AI 领域的应用。

## 1.7 这个理念从何而来？

Code as Proxy 并非凭空出现。它的思想根源可以追溯到多个领域：

### 1.7.1 Palantir 的 AIP 架构

Palantir 是这一理念最成功的商业实践者。其 AIP（AI Platform）的核心设计就是：LLM 操作 Ontology（业务本体），而非原始数据。这使得 Palantir AIP 成为**率先获得 IL6（机密级，SECRET）部署资质的商业产品**。

### 1.7.2 联邦学习的启示

联邦学习（Federated Learning）的核心思想是"数据不动模型动"——模型被送到数据所在地训练，而非数据被集中到一处。Code as Proxy 将这一思想从训练阶段扩展到了推理阶段："数据不动代码动"。

### 1.7.3 存储过程的回归

数据库存储过程（Stored Procedure）的设计哲学是：将计算逻辑送到数据旁边执行，而非将数据拉到计算节点。Code as Proxy 是这一古老智慧在 AI 时代的复兴。

### 1.7.4 零信任安全架构

零信任（Zero Trust）的核心原则是"永不信任，始终验证"。Code as Proxy 将零信任从网络层扩展到了 AI 层：不信任 AI 会安全处理数据，所以不给它数据。

## 1.8 适用边界

Code as Proxy 不是银弹。它有明确的适用边界：

### ✅ 适用场景

- **结构化数据分析**：SQL 查询、统计分析、报表生成
- **数据质量检测**：基于 Schema 生成校验规则
- **特征工程**：根据字段描述生成特征提取代码
- **业务流程自动化**：基于 Ontology 生成操作指令
- **合规报告**：生成报告代码，敏感字段在本地脱敏

### ❌ 不适用场景

- **图像/视频理解**：AI 必须"看到"图像内容才能分析
- **自然语言理解**：AI 必须读到文本内容才能理解语义
- **异常检测（基于内容）**：需要 AI 理解数据的具体值
- **个性化推荐**：需要 AI 理解用户的具体行为数据

对于不适用的场景，需要结合其他安全策略（如差分隐私、同态加密、可信执行环境）来补充。

## 本章小结

| 要点 | 说明 |
|------|------|
| **核心矛盾** | AI 需要数据才能工作，但数据不能离开安全边界 |
| **Code as Proxy** | 让 AI 写代码操作数据，而非直接接触数据 |
| **四层架构** | 业务语义层 → 代理生成层 → 受控执行层 → 结果摘要层 |
| **本质区别** | 从"信任 AI 处理数据"到"AI 根本看不到数据" |
| **适用边界** | 适用于结构化数据操作，不适用于需要理解内容的场景 |

> **核心洞察**：Code as Proxy 不是一种安全技术，而是一种架构哲学。它重新定义了 AI 与数据的关系——从"AI 消费数据"变为"AI 生成操作数据的代理"。这一范式转换，让"AI 能力"和"数据安全"从零和博弈变为正和博弈。
