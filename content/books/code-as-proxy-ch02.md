# 第 2 章：理论基础——从信息论到零信任

> 选自《Code as Proxy — AI 时代的数据安全架构》

## 2.1 信息论视角：AI 真的需要看到所有数据吗？

Code as Proxy 的理论根基之一来自信息论。核心问题是：**AI 完成一个任务，到底需要多少信息？**

### 2.1.1 最小充分统计量（Minimal Sufficient Statistic）

在统计学中，**充分统计量**是指一个统计量包含了样本中关于参数的所有信息。换言之，给定充分统计量后，原始数据不再提供额外信息。

例如，要估计正态分布的均值和方差，你只需要样本均值和样本方差——不需要看到每一个原始数据点。

Code as Proxy 的信息论基础正是如此：

```
原始数据 D = {x₁, x₂, ..., xₙ}
充分统计量 T(D) = {Schema, 聚合统计, 分布描述}

对于大多数 AI 任务：
  I(θ; T(D)) = I(θ; D)
  即：充分统计量包含了 AI 完成任务所需的全部信息
```

**Schema 就是一种充分统计量。** 它告诉 AI 数据的结构（字段名、类型、约束、关系），AI 据此生成操作代码。对于代码生成任务，Schema 已经是"充分"的——原始数据值不提供额外的代码生成信息。

### 2.1.2 信息瓶颈理论（Information Bottleneck）

Tishby 等人（2000）提出的信息瓶颈理论为 Code as Proxy 提供了更深层的理论支撑：

> 在输入 X 和目标 Y 之间，存在一个最优的压缩表示 T，它保留了 X 中关于 Y 的所有相关信息，同时丢弃了无关信息。

```
min I(X; T) - β · I(T; Y)
    ↑              ↑
  最小化信息暴露   最大化任务相关性
```

在 Code as Proxy 的语境下：
- **X** = 原始数据（包含敏感信息）
- **Y** = AI 需要完成的任务（生成代码/分析报告）
- **T** = Schema + 元数据（压缩表示）

信息瓶颈告诉我们：**存在一个最优的信息压缩点，在这个点上，AI 获得了完成任务所需的全部信息，同时暴露的敏感信息最少。** Schema 正是这个压缩点的一个近似。

### 2.1.3 差分隐私的互补视角

差分隐私（Differential Privacy, Dwork et al., 2006）从另一个角度支撑了 Code as Proxy：

差分隐私的核心思想是：在数据中添加噪声，使得任何单个数据点的存在与否不会显著影响输出结果。其数学定义为：

```
Pr[M(D) ∈ S] ≤ e^ε · Pr[M(D') ∈ S]
其中 D 和 D' 是仅差一条记录的相邻数据集
```

Code as Proxy 可以看作差分隐私的一种**极端形式**：不是在数据中添加噪声，而是**根本不给 AI 数据**。AI 只看到 Schema（ε → 0 的极限情况），隐私保护达到理论最优。

当然，这种极端形式有代价——AI 无法完成需要理解数据内容的任务。但对于代码生成、数据管道构建、报表生成等任务，这种代价是可接受的。

## 2.2 最小权限原则的 AI 扩展

### 2.2.1 经典最小权限原则

最小权限原则（Principle of Least Privilege, Saltzer & Schroeder, 1975）是信息安全的基石之一：

> 每个主体（用户、进程、程序）应该只被授予完成其任务所需的最小权限集合。

在传统系统中，这意味着：
- 数据库用户只能访问其需要的表
- 应用程序只能调用其需要的 API
- 操作系统进程只能访问其需要的文件

### 2.2.2 AI 时代的权限困境

但 AI 打破了传统的权限模型。一个 LLM 的"权限需求"是模糊的、动态的、难以预定义的：

```
传统应用：
  权限 = {读取 users 表, 写入 orders 表, 调用 payment API}
  → 静态、可枚举、可审计

AI Agent：
  权限 = {理解任意自然语言指令, 生成任意代码, 调用任意工具}
  → 动态、不可枚举、难以审计
```

传统的 RBAC（基于角色的访问控制）和 ABAC（基于属性的访问控制）在面对 AI 时力不从心，因为 AI 的行为空间是开放的。

### 2.2.3 Code as Proxy 的权限重构

Code as Proxy 通过**将权限从 AI 转移到代码**来解决这个困境：

```
传统模式：
  AI 拥有权限 → AI 直接访问数据 → 权限难以约束

Code as Proxy：
  AI 无权限 → AI 生成代码 → 代码拥有受限权限 → 代码访问数据
                              ↑
                    代码的权限可以被精确定义和审计
```

这是一种**权限代理（Permission Delegation）**模式：

| 维度 | AI 的权限 | 代码的权限 |
|------|----------|-----------|
| 范围 | 仅 Schema 读取 | 由 ACL 精确定义 |
| 粒度 | 粗粒度（全有或全无） | 细粒度（字段级/行级） |
| 时效 | 无限制 | 可设置过期时间 |
| 审计 | 难以审计 AI 的"思考过程" | 代码完全可审计 |
| 撤销 | 无法撤销 AI 已"看到"的数据 | 可以撤销代码的执行权限 |

**核心洞察**：AI 的权限问题之所以难解，是因为我们试图约束一个行为空间无限的主体。Code as Proxy 的解法是：不约束 AI，而是约束 AI 的产出物（代码）。代码的行为空间是有限的、可分析的、可审计的。

## 2.3 数据主权理论

### 2.3.1 数据主权的三层含义

数据主权（Data Sovereignty）在 AI 时代有三层含义：

**第一层：地理主权（Data Residency）**
数据必须存储在特定的地理区域内，受该区域法律管辖。GDPR、中国《数据安全法》、《个人信息保护法》都有明确的数据本地化要求。

**第二层：控制主权（Data Control）**
数据所有者对数据的使用方式拥有完全控制权。包括谁可以访问、如何处理、保留多久、是否可以转移。

**第三层：认知主权（Data Cognition Sovereignty）**
这是 AI 时代新增的维度：**谁可以"理解"你的数据？** 当数据被发送给外部 LLM 时，即使 LLM 提供商承诺不存储数据，LLM 在推理过程中已经"理解"了数据的内容。这种"认知"是否构成对数据主权的侵犯？

### 2.3.2 Code as Proxy 与数据主权

Code as Proxy 在三个层面都保护了数据主权：

| 主权层面 | 传统 AI 模式 | Code as Proxy |
|---------|-------------|---------------|
| 地理主权 | 数据传输到外部服务器 | 数据不出域 ✅ |
| 控制主权 | 数据使用方式由 AI 提供商决定 | 数据使用由本地代码控制 ✅ |
| 认知主权 | AI "理解"了数据内容 | AI 只"理解"数据结构 ✅ |

特别是**认知主权**——这是 Code as Proxy 独有的优势。即使使用了端到端加密、零知识证明等技术，传统模式下 AI 在推理时仍然需要"理解"明文数据。Code as Proxy 从根本上避免了这个问题：AI 理解的是 Schema，不是数据。

### 2.3.3 法律框架的支撑

全球主要数据保护法规都在向 Code as Proxy 友好的方向演进：

- **GDPR（欧盟）**：数据最小化原则（Art. 5(1)(c)）——只收集和处理完成目的所需的最少数据。Code as Proxy 天然满足这一原则。
- **中国《数据安全法》**：数据处理活动应当在必要范围内进行。Schema 是"必要范围"的最小化表达。
- **美国 CCPA/CPRA**：消费者有权知道其数据被如何使用。Code as Proxy 下，数据根本没有被发送给 AI，合规审计大幅简化。
- **行业法规（HIPAA/PCI-DSS/SOX）**：对数据传输和存储有严格要求。Code as Proxy 下数据不移动，天然满足。

## 2.4 联邦计算范式

### 2.4.1 从联邦学习到联邦推理

联邦学习（Federated Learning, McMahan et al., 2017）开创了"数据不动模型动"的范式：

```
联邦学习：
  模型 ──→ 数据节点 ──→ 本地训练 ──→ 梯度上传 ──→ 聚合更新
  （数据留在本地，只有梯度/参数离开）
```

Code as Proxy 将这一范式从训练扩展到推理：

```
联邦推理（Code as Proxy）：
  Schema ──→ AI ──→ 生成代码 ──→ 代码下发到数据节点 ──→ 本地执行 ──→ 结果摘要
  （数据留在本地，只有代码和摘要离开）
```

两者的共同哲学是：**计算应该去数据所在的地方，而不是数据去计算所在的地方。**

### 2.4.2 与同态加密的对比

同态加密（Homomorphic Encryption）允许在加密数据上直接计算，无需解密。理论上完美，但实践中面临巨大挑战：

| 维度 | 同态加密 | Code as Proxy |
|------|---------|---------------|
| 计算开销 | 极高（10,000-1,000,000x） | 几乎无额外开销 |
| 支持的操作 | 有限（加法/乘法） | 任意代码 |
| 实现复杂度 | 极高 | 中等 |
| 成熟度 | 学术研究为主 | 已有商业实践（Palantir） |
| 适用场景 | 简单统计计算 | 复杂业务逻辑 |

Code as Proxy 的实用性远超同态加密，因为它不需要在密文上计算——它根本不需要数据。

### 2.4.3 与可信执行环境（TEE）的对比

可信执行环境（Trusted Execution Environment, 如 Intel SGX、ARM TrustZone）提供硬件级的安全隔离：

| 维度 | TEE | Code as Proxy |
|------|-----|---------------|
| 安全保证 | 硬件级隔离 | 架构级隔离 |
| 硬件依赖 | 需要特定硬件 | 无硬件依赖 |
| 性能影响 | 中等（10-30% 开销） | 几乎无开销 |
| 攻击面 | 侧信道攻击（Spectre/Meltdown） | 代码注入攻击 |
| 部署难度 | 高（需要硬件支持） | 中（纯软件方案） |

在实践中，Code as Proxy 和 TEE 可以互补：代码在 TEE 中执行，提供双重保护。

## 2.5 学术研究综述

Code as Proxy 的理念在学术界有多个相关研究方向：

### 2.5.1 Schema-Aware Code Generation

**NL2SQL 领域**是 Code as Proxy 最成熟的学术实践。AI 根据数据库 Schema 生成 SQL 查询，无需接触实际数据：

- **Spider**（Yu et al., 2018）：跨数据库 NL2SQL 基准，AI 仅根据 Schema 生成 SQL
- **BIRD**（Li et al., 2023）：大规模 NL2SQL 基准，引入数据库值的统计描述作为辅助信息
- **DIN-SQL**（Pourreza & Rafiei, 2023）：将 NL2SQL 分解为 Schema 链接 → 查询分类 → SQL 生成的流水线
- **DAIL-SQL**（Gao et al., 2023）：利用 LLM 的 In-Context Learning 能力，仅通过 Schema + 少量示例生成高质量 SQL

这些工作证明了：**AI 仅凭 Schema 就能生成高质量的数据操作代码。**

### 2.5.2 Secure Code Generation

安全代码生成领域关注 AI 生成代码的安全性：

- **Pearce et al.（2022）**："Asleep at the Keyboard"——分析 GitHub Copilot 生成代码中的安全漏洞
- **He & Vechev（2023）**：提出 LLM 生成代码的形式化验证框架
- **Tony et al.（2023）**：LLMSecEval——评估 LLM 生成安全代码的能力
- **Sandoval et al.（2023）**：研究 AI 辅助编码对代码安全性的影响

### 2.5.3 Privacy-Preserving AI

隐私保护 AI 领域与 Code as Proxy 高度相关：

- **Abadi et al.（2016）**：深度学习中的差分隐私——DP-SGD 算法
- **Bonawitz et al.（2019）**：大规模联邦学习的工程实践（Google）
- **Kairouz et al.（2021）**：联邦学习的进展与开放问题综述
- **Li et al.（2024）**：LLM 时代的隐私保护——从训练到推理的全链路

### 2.5.4 Ontology-Driven Systems

本体论驱动的系统是 Code as Proxy 的业务语义层基础：

- **Guarino（1998）**：形式本体论的哲学基础
- **Noy & McGuinness（2001）**：本体论开发 101——方法论指南
- **Hogan et al.（2021）**：知识图谱综述——从理论到实践
- **Pan et al.（2024）**：大模型与知识图谱的融合——LLM + KG 的协同范式
