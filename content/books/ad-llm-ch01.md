# 第 1 章：数据——从采集到合成的全链路

> 选自《自动驾驶大模型深度研究》

## 1.1 为什么数据是自动驾驶的「第一生产力」

自动驾驶系统的能力上限由数据决定。与 NLP 领域可以从互联网爬取万亿 token 不同，驾驶数据的获取成本极高：

| 数据类型 | 采集成本 | 标注成本 | 规模量级 |
|---------|---------|---------|---------|
| 文本（Common Crawl） | 接近免费 | 不需要 | 15T+ tokens |
| 图像（ImageNet） | 低 | $0.01/张 | 千万级 |
| 驾驶视频（nuScenes） | $50+/小时 | $100+/帧 | 万级场景 |
| 激光雷达点云 | $200+/小时 | $500+/帧 | 千级场景 |

一辆搭载全套传感器的测试车每天产生 **5-20TB** 原始数据，但其中仅 0.1% 包含有价值的长尾场景（corner cases）。这构成了自动驾驶数据工程的核心挑战。

### 数据同步与校准

多传感器数据必须在时间和空间上精确对齐：

- **时间同步**：PTP（Precision Time Protocol），精度 < 1μs，所有传感器共享同一时钟源
- **空间标定**：激光雷达-相机外参标定（通常使用棋盘格靶标），精度要求 < 1cm
- **在线标定**：运行时持续修正因振动导致的外参漂移

```python
# 传感器融合的时间对齐（简化示例）
def temporal_align(lidar_frames, camera_frames, max_dt_ms=50):
    """基于时间戳的最近邻对齐"""
    aligned_pairs = []
    for lidar_f in lidar_frames:
        t_lidar = lidar_f.timestamp
        # 找最近的相机帧
        nearest_cam = min(camera_frames, 
                         key=lambda c: abs(c.timestamp - t_lidar))
        dt = abs(nearest_cam.timestamp - t_lidar)
        if dt < max_dt_ms:
            aligned_pairs.append((lidar_f, nearest_cam))
    return aligned_pairs
```

## 1.3 数据标注体系

### 标注类型与精度要求

| 标注任务 | 精度要求 | 工具 | 效率 |
|---------|---------|------|------|
| 3D 物体检测 | IoU > 0.7, 位置误差 < 10cm | 点云标注工具 (CVAT, Labelbox) | 5-10 帧/人/小时 |
| 语义分割 | mIoU > 0.95 | 多边形/像素标注 | 2-4 帧/人/小时 |
| 车道线 | 横向误差 < 5cm | 样条曲线标注 | 10-20 帧/人/小时 |
| 行为预测轨迹 | ADE < 0.5m @3s | 轨迹序列标注 | 20-30 个场景/人/小时 |

### 自动标注（Auto-Labeling）

人工标注成本过高催生了自动标注技术，其核心思路是**用大模型标注训练小模型**：

```
离线大模型（高精度, 慢）              在线小模型（低延迟, 快）
  PointPillars++ (3D检测)               BEVFusion-Lite
  + SAM 2 (分割)          ──标注──→     + FastBEV
  + GPT-4V (场景描述)     ──训练──→     + TinyLLaVA
  
  精度: mAP 0.85                        精度: mAP 0.78
  延迟: 500ms/帧                        延迟: 30ms/帧
```

**关键技术**：
1. **置信度过滤**：只保留模型置信度 > 0.9 的自动标注结果
2. **时间一致性验证**：连续帧的标注结果应保持物理一致性（速度、加速度合理）
3. **人工抽检**：随机抽取 5-10% 自动标注结果人工审核

## 1.4 数据合成：虚拟数据工厂

### 为什么需要合成数据？

真实世界的长尾场景极其稀少：
- 暴雨中行人突然横穿 → 可能需要 100 万公里才能遇到一次
- 施工区域异形障碍物 → 每个城市的情况都不同
- 传感器退化（镜头起雾、激光雷达脏污） → 难以系统性采集

合成数据可以按需生成任意场景，成本仅为真实采集的 1/100。

### 主流合成技术

**1. 仿真引擎渲染（Explicit）**

```
场景描述 → 仿真引擎 (CARLA/NVIDIA Omniverse) → 渲染多模态数据
           ↓
      物理引擎 (PhysX)
      光照模型 (Ray Tracing)  → 相机图像 + 激光雷达点云 + 标注
      交通流模拟 (SUMO)
```

优势：完全可控，标注零成本。局限：域间隙（sim-to-real gap）显著。

**2. 神经辐射场/高斯泼溅（Neural Rendering）**

```
真实场景扫描 → 3D Gaussian Splatting → 新视角渲染
              ↓ 编辑
         修改天气 / 插入物体 / 改变光照 → 合成数据 + 自动标注
```

代表工作：NVIDIA NeRF-based scene editing, UniSim (Waabi)。优势：与真实数据的分布更接近。

**3. 生成式模型（Generative）**

```
文本描述 → 扩散模型 (MagicDrive / DriveDreamer) → 多视角驾驶视频
                                                   ↓ 
                                              BEV 布局 → 3D 标注
```

2025-2026 年前沿：用视频扩散模型直接生成连续多帧驾驶场景，结合 BEV 控制保证场景一致性。

## 1.5 数据增强策略

| 增强方法 | 应用 | 效果 |
|---------|------|------|
| 几何变换（旋转/缩放/翻转） | 图像/点云 | +2-5% mAP |
| 混合增强（CutMix/MixUp） | 图像 | +1-3% mAP |
| 点云增强（GT-Paste） | 3D 检测 | +3-5% mAP |
| 域随机化（颜色/纹理/天气） | 仿真数据 | 缩小 sim-to-real gap |
| 对抗增强（PGD/AutoAugment） | 鲁棒性 | 提升 corner case 表现 |

**GT-Paste** 是自动驾驶中最有效的增强技术之一：从数据库中取出标注好的 3D 物体（行人、车辆），直接粘贴到新场景的点云中，并检查物理合理性（不穿墙、不悬浮）。

## 1.6 数据治理与合规

### 隐私保护

自动驾驶数据包含大量隐私信息（人脸、车牌、位置轨迹），需要：
- **自动脱敏**：人脸模糊（RetinaFace + GaussianBlur）、车牌打码
- **差分隐私**：轨迹数据添加拉普拉斯噪声
- **联邦学习**：数据不出本地，只共享模型梯度

### 数据版本管理

```
DVC (Data Version Control) + Git
  └── 管理 TB 级数据集的版本
  └── S3/GCS 远程存储
  └── pipeline 追踪（采集→标注→增强→训练）
```
