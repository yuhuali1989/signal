# 第 7 章：未来演进——从代理代码到自主安全系统

> 选自《Code as Proxy — AI 时代的数据安全架构》

## 7.1 Code as Proxy 的演进方向

Code as Proxy 在 2026 年仍处于早期阶段。随着 AI 能力的指数级增长和数据安全需求的持续升级，这一架构范式将沿着多个方向演进。

```
演进路线图：

2024-2025  Code as Proxy 1.0
           ├── Schema-Driven SQL/Code Generation
           ├── 基础沙箱执行
           └── 手动 Schema 维护

2026-2027  Code as Proxy 2.0
           ├── Ontology-Grounded Action Generation
           ├── AI Agent 原生集成
           ├── 自动 Schema 发现和增强
           └── 多模型交叉验证

2028-2029  Code as Proxy 3.0
           ├── 自进化 Ontology
           ├── 联邦代码网络
           ├── 形式化验证
           └── 自主安全系统

2030+      Code as Proxy 4.0
           ├── 量子安全 Code as Proxy
           ├── 通用 AI 安全架构
           └── AI 与数据的新型关系
```

## 7.2 Agent 原生架构

### 7.2.1 从工具到基础设施

当前的 Code as Proxy 主要作为 AI 的一个"工具"存在——AI Agent 在需要访问数据时调用 Code as Proxy。未来，Code as Proxy 将成为 AI Agent 的**基础设施层**：

```
当前：Agent 调用 Code as Proxy 工具
  Agent ──→ [Tool: secure_data_query] ──→ Code as Proxy ──→ 结果

未来：Agent 原生运行在 Code as Proxy 之上
  ┌─────────────────────────────────────────┐
  │         Code as Proxy Runtime            │
  │  ┌─────────────────────────────────┐    │
  │  │          AI Agent               │    │
  │  │  - 所有数据访问自动经过 CaP     │    │
  │  │  - 所有代码生成自动经过验证     │    │
  │  │  - 所有结果自动经过摘要         │    │
  │  │  - Agent 无法绕过安全层         │    │
  │  └─────────────────────────────────┘    │
  └─────────────────────────────────────────┘
```

这意味着 Code as Proxy 不再是一个可选的安全插件，而是 Agent 运行环境的一部分——就像操作系统的内存保护机制一样，应用程序无法绕过。

### 7.2.2 Multi-Agent Code as Proxy

在多 Agent 协作场景中，Code as Proxy 需要处理 Agent 间的数据隔离：

```
┌──────────────────────────────────────────────┐
│              Multi-Agent CaP Runtime          │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ Agent A   │  │ Agent B   │  │ Agent C   │  │
│  │ (销售分析) │  │ (风控)    │  │ (客服)    │  │
│  │           │  │           │  │           │  │
│  │ 可见:     │  │ 可见:     │  │ 可见:     │  │
│  │ 销售Schema│  │ 交易Schema│  │ 工单Schema│  │
│  │ 客户Schema│  │ 风险Schema│  │ 客户Schema│  │
│  │ (部分)    │  │           │  │ (部分)    │  │
│  └──────────┘  └──────────┘  └──────────┘  │
│       │              │              │        │
│       ▼              ▼              ▼        │
│  ┌──────────────────────────────────────┐   │
│  │     Ontology-Based Access Control     │   │
│  │  每个 Agent 只能看到其角色对应的       │   │
│  │  Ontology 子集，生成的代码只能访问     │   │
│  │  其权限范围内的数据                    │   │
│  └──────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
```

### 7.2.3 Agent-to-Agent 安全通信

当 Agent A 需要 Agent B 的数据时，不是直接共享数据，而是通过 Code as Proxy 进行安全的间接通信：

```
Agent A 需要风控数据来辅助销售决策：

传统方式（❌）：
  Agent A ──→ "给我客户 X 的风险评分" ──→ Agent B
  Agent B ──→ "客户 X 风险评分 0.73" ──→ Agent A
  问题：具体风险评分泄露给了销售 Agent

Code as Proxy 方式（✅）：
  Agent A ──→ "这批客户中，哪些风险可接受？" ──→ CaP Router
  CaP Router ──→ 生成筛选代码 ──→ 风控数据库执行
  CaP Router ──→ "47 个客户风险可接受，23 个需要额外审查" ──→ Agent A
  Agent A 只知道结果，不知道具体风险评分
```

## 7.3 自进化 Ontology

### 7.3.1 当前的痛点

Ontology 的构建和维护是 Code as Proxy 最大的成本中心。当前主要依赖人工（FDE）进行：

- 业务理解 → 手动建模
- 业务变化 → 手动更新
- 新数据源 → 手动集成

### 7.3.2 AI 辅助 Ontology 构建

未来，AI 将参与 Ontology 的自动构建和维护：

```
自进化 Ontology 流程：

1. 自动发现（Auto-Discovery）
   AI 分析数据库 Schema + 查询日志 + 业务文档
   → 自动推荐实体、关系和约束
   → 人工审核确认

2. 自动增强（Auto-Enhancement）
   AI 分析用户查询模式
   → 发现 Schema 中缺失的语义描述
   → 自动补充字段描述和统计信息
   → 人工审核确认

3. 自动演进（Auto-Evolution）
   AI 监控数据变化和业务流程变化
   → 检测 Ontology 与实际数据的偏差
   → 建议 Ontology 更新
   → 人工审核确认

4. 质量反馈（Quality Feedback）
   AI 生成的代码执行失败或结果异常
   → 分析失败原因
   → 反馈到 Ontology 改进
   → 形成闭环
```

注意：**人工审核始终是必要的。** Ontology 定义了 AI 的行为边界，如果 Ontology 被错误修改，AI 可能生成不安全的代码。

### 7.3.3 跨组织 Ontology 标准

随着 Code as Proxy 的普及，行业级的 Ontology 标准将出现：

| 行业 | 标准 Ontology | 覆盖范围 | 复用率 |
|------|-------------|---------|--------|
| 金融 | FIBO (Financial Industry Business Ontology) | 金融产品、交易、风控 | ~70% |
| 医疗 | SNOMED CT + HL7 FHIR | 疾病、药物、检验、流程 | ~80% |
| 制造 | ISA-95 + OPC UA | 设备、产线、工单、质检 | ~75% |
| 自动驾驶 | OpenSCENARIO + ASAM | 场景、传感器、标注 | ~65% |

标准 Ontology 将大幅降低 Code as Proxy 的实施成本——新客户可以基于行业标准 Ontology 快速启动，只需定制 20-30% 的业务特殊性。

## 7.4 联邦代码网络

### 7.4.1 从点对点到网络

当前的 Code as Proxy 是点对点的：一个 AI 生成代码，在一个数据节点执行。未来将演进为**联邦代码网络**——多个 AI 和多个数据节点组成的分布式网络：

```
联邦代码网络：

  ┌─────┐    ┌─────┐    ┌─────┐
  │AI-1 │    │AI-2 │    │AI-3 │
  └──┬──┘    └──┬──┘    └──┬──┘
     │          │          │
     ▼          ▼          ▼
  ┌──────────────────────────────┐
  │     Code Routing Network      │
  │  (代码路由网络)                │
  │  - 根据数据位置路由代码        │
  │  - 跨节点结果聚合             │
  │  - 全局安全策略执行            │
  └──────────────────────────────┘
     │          │          │
     ▼          ▼          ▼
  ┌─────┐    ┌─────┐    ┌─────┐
  │数据A │    │数据B │    │数据C │
  │(银行)│    │(医院)│    │(政府)│
  └─────┘    └─────┘    └─────┘
```

### 7.4.2 跨组织安全协作

联邦代码网络使得跨组织的数据协作成为可能，而无需共享任何原始数据：

**场景：反洗钱跨行协作**

```
银行 A、B、C 需要联合检测跨行洗钱模式，
但不能共享客户交易数据。

传统方式（❌）：
  各银行将数据上传到中央平台 → 隐私风险极高

联邦代码网络（✅）：
  1. 各银行提供交易 Schema（不含数据）
  2. AI 根据联合 Schema 生成检测代码
  3. 代码分发到各银行本地执行
  4. 各银行返回聚合摘要（如"可疑模式数量"）
  5. 中央节点聚合摘要，识别跨行模式
  
  数据全程不出各银行边界。
```

## 7.5 形式化验证

### 7.5.1 为什么需要形式化验证？

当前的代码安全验证主要依赖模式匹配和沙箱隔离——这是"尽力而为"的防御。未来，对于高安全场景（国防、金融核心系统），需要**形式化验证**——数学证明代码的安全性。

### 7.5.2 可验证代码生成

```
未来的 Code as Proxy 流程：

Schema ──→ AI ──→ 生成代码 + 安全性证明
                        │
                        ▼
              形式化验证器（如 Z3 / Coq）
                        │
                   ┌────┴────┐
                   │         │
                验证通过    验证失败
                   │         │
                   ▼         ▼
                执行代码   拒绝执行
                           + 反馈给 AI 重新生成
```

AI 不仅生成代码，还生成代码的安全性证明（如"此代码不会访问 PII 字段"、"此代码的输出不包含超过 K 条记录的明细"）。形式化验证器检查证明的正确性。

### 7.5.3 研究前沿

- **Verified Code Generation**（He & Vechev, 2023）：LLM 生成代码 + 形式化规约
- **Proof-Carrying Code**（Necula, 1997）：代码携带安全性证明
- **Certified Compilation**（Leroy, 2009, CompCert）：编译过程保持安全性质

将这些技术与 Code as Proxy 结合，可以实现**数学可证明的数据安全**。

## 7.6 自主安全系统

### 7.6.1 从被动防御到主动安全

当前的安全系统是被动的——检测攻击、响应事件。未来的 Code as Proxy 将具备**自主安全能力**：

```
自主安全系统：

  ┌─────────────────────────────────────┐
  │         Security AI Agent            │
  │                                     │
  │  持续监控：                          │
  │  - 代码生成模式是否异常？            │
  │  - 查询模式是否存在推断攻击？        │
  │  - Schema 是否被篡改？              │
  │  - 沙箱是否被突破？                 │
  │                                     │
  │  自主响应：                          │
  │  - 检测到异常 → 自动限流             │
  │  - 检测到攻击 → 自动阻断 + 告警     │
  │  - 检测到漏洞 → 自动修补 + 验证     │
  │  - 检测到新威胁 → 自动更新规则      │
  └─────────────────────────────────────┘
```

### 7.6.2 对抗性安全

未来的安全系统将采用**对抗性方法**——用 AI 攻击 AI，持续发现和修复漏洞：

```
Red Team AI（攻击方）：
  - 尝试通过 Prompt Injection 生成恶意代码
  - 尝试通过查询模式推断敏感数据
  - 尝试绕过沙箱限制
  - 尝试篡改 Schema

Blue Team AI（防御方）：
  - 检测和阻断 Red Team 的攻击
  - 分析攻击模式，更新防御规则
  - 修复发现的漏洞
  - 生成安全报告

持续对抗 → 安全能力持续提升
```

## 7.7 AI 与数据关系的终极形态

### 7.7.1 三种范式

回顾 AI 与数据关系的演进，我们可以看到三种范式：

```
范式 1：AI 消费数据（2020-2024）
  数据 ──→ AI ──→ 结果
  AI 直接接触原始数据
  安全性：低

范式 2：AI 操作数据的代理（2024-2028）
  Schema ──→ AI ──→ Code ──→ 数据 ──→ 摘要
  AI 通过代码间接操作数据
  安全性：高（Code as Proxy）

范式 3：AI 与数据的共生（2028+）
  AI 和数据在同一个安全边界内共存
  AI 是数据系统的一部分，而非外部消费者
  安全性：最高（数据原生 AI）
```

### 7.7.2 数据原生 AI

范式 3 的终极形态是**数据原生 AI**——AI 不是一个外部服务，而是数据系统的内置能力：

```
数据原生 AI：

  ┌─────────────────────────────────────┐
  │          数据系统                     │
  │                                     │
  │  ┌──────────┐  ┌──────────────┐    │
  │  │ 数据存储  │  │ 内置 AI 引擎  │    │
  │  │          │←→│              │    │
  │  │ 数据不出  │  │ AI 不出      │    │
  │  │ 系统边界  │  │ 系统边界     │    │
  │  └──────────┘  └──────────────┘    │
  │                                     │
  │  AI 和数据在同一个安全边界内         │
  │  无需 Code as Proxy 的间接层         │
  │  但安全性由系统架构保证              │
  └─────────────────────────────────────┘
```

这就像数据库的存储过程——计算和数据在同一个系统内，无需数据传输。但与存储过程不同的是，数据原生 AI 具备自然语言理解和推理能力。

Snowflake Cortex、Databricks AI Functions 已经在朝这个方向演进——让 AI 在数据仓库内部运行，数据不需要离开数据仓库。

## 7.8 给实践者的建议

### 7.8.1 现在就开始

不要等待完美的解决方案。Code as Proxy 1.0（Schema-Driven Code Generation + 基础沙箱）已经足够实用，可以立即开始：

1. **从 Schema 建设开始**：为你最重要的数据资产构建增强 Schema
2. **选择低风险场景试点**：数据分析报表是最好的起点
3. **逐步加强安全**：从基础沙箱开始，逐步添加差分隐私、形式化验证等高级特性

### 7.8.2 投资 Ontology

Ontology 是 Code as Proxy 的长期竞争力。投资 Ontology 建设的回报是指数级的：

- **第 1 个月**：Ontology 覆盖核心实体，AI 可以完成基础查询
- **第 6 个月**：Ontology 覆盖主要业务流程，AI 可以辅助决策
- **第 12 个月**：Ontology 成为业务运转的"数字神经系统"，不可替代

### 7.8.3 关注标准化

行业 Ontology 标准（FIBO、FHIR、ISA-95）将大幅降低实施成本。关注并参与标准化进程，可以获得先发优势。

### 7.8.4 培养复合人才

Code as Proxy 需要同时理解 AI、安全和业务的复合人才。这类人才极其稀缺，需要有意识地培养：

| 能力 | 重要性 | 培养方式 |
|------|--------|---------|
| AI/LLM 工程 | ⭐⭐⭐⭐⭐ | Prompt Engineering + Agent 开发 |
| 安全工程 | ⭐⭐⭐⭐ | 沙箱技术 + 威胁建模 + 合规 |
| 业务理解 | ⭐⭐⭐⭐⭐ | 驻场实践（FDE 模式） |
| 数据工程 | ⭐⭐⭐⭐ | Schema 设计 + 数据治理 |
| 形式化方法 | ⭐⭐⭐ | 程序验证 + 类型理论（长期） |

## 全书总结

七章走完，让我们回到最初的问题：

> **AI 越强大，它需要接触的数据越多；数据越敏感，你越不敢让 AI 碰它。**

Code as Proxy 给出的答案是：

> **不要让 AI 碰数据。让 AI 写代码去碰数据。**

这个看似简单的思路转换，背后是：
- **信息论**的最小充分统计量
- **安全理论**的最小权限原则
- **数据主权**的三层保护
- **联邦计算**的"数据不动代码动"
- **Palantir** 的 $260B 商业验证

Code as Proxy 不是银弹，但它是 AI 时代数据安全的**最佳实践起点**。

从今天开始，为你的数据构建 Schema，让 AI 写代码，而不是碰数据。
