# 第 4 章：推理优化——KV Cache、量化与投机采样

> 选自《大语言模型从入门到前沿》

## 4.1 LLM 推理的两阶段与性能瓶颈

LLM 推理分为两个截然不同的阶段：

| 阶段 | 名称 | 特点 | 瓶颈 | GPU 利用率 |
|------|------|------|------|-----------|
| Prefill | 预填充 | 并行处理所有输入 token | **计算瓶颈**（compute-bound） | 60-80% |
| Decode | 逐步解码 | 每次只生成 1 个 token | **访存瓶颈**（memory-bound） | < 5% |

### 4.1.1 为什么 Decode 阶段效率如此低？

Decode 阶段的核心矛盾：每生成一个 token，需要加载整个模型权重（几十 GB），但只做极少计算。这导致 GPU 利用率极低。

用一个具体的数字感受：Llama 3 70B 模型 FP16 权重约 140GB，存储在 HBM（高带宽内存）中。H100 的 HBM 带宽约 3.35 TB/s，意味着加载一次模型需要约 42ms。而这 42ms 的加载只为一个 token 做几百 MFLOPS 的计算——相比 H100 的 990 TFLOPS 理论峰值，利用率不到 0.1%。

### 4.1.2 核心性能指标

推理系统的性能指标是多维度的：

```python
# 推理性能指标体系
metrics = {
    "延迟指标": {
        "TTFT": "Time To First Token（首 token 延迟）",
        "TPOT": "Time Per Output Token（逐 token 延迟）",
        "E2E_latency": "端到端延迟 = TTFT + TPOT × output_len",
    },
    "吞吐指标": {
        "tokens_per_second": "每秒生成 token 数",
        "requests_per_second": "每秒完成请求数",
        "goodput": "满足 SLO 的有效吞吐（DistServe 提出）",
    },
    "效率指标": {
        "MBU": "Model Bandwidth Utilization = 实际/理论带宽",
        "MFU": "Model FLOPS Utilization = 实际/理论算力",
        "cost_per_token": "每 token 成本（$/M tokens）",
    },
}
```

**Goodput** 是 DistServe (OSDI 2024) 提出的关键指标：不是所有生成的 token 都算数，只有满足延迟 SLO（Service Level Objective）的请求才计入 Goodput。这与传统的吞吐率有本质区别。

### 4.1.3 Roofline 模型分析

推理的计算/访存瓶颈可以用 Roofline 模型量化：

$$\text{操作强度 (OI)} = \frac{\text{FLOPS}}{\text{Bytes transferred}}$$

当 OI < 硬件的"平衡点"时，受访存限制；反之受计算限制。

```
H100 Roofline (FP16):
  峰值算力: 990 TFLOPS
  HBM 带宽: 3.35 TB/s
  平衡点 OI: 990 / 3.35 ≈ 295 FLOP/Byte

Prefill OI ≈ 2×d_model ≈ 16384  >> 295  → 计算瓶颈 ✅
Decode OI ≈ 2 (batch=1)         << 295  → 访存瓶颈 ✅
Decode OI ≈ 2×batch_size         → batch ≥ 148 时转为计算瓶颈
```

**关键洞察**：通过增大 batch size（continuous batching），可以将 Decode 阶段从访存瓶颈推向计算瓶颈，这是 vLLM 等框架的核心优化思路。

## 4.2 KV Cache：用空间换时间

### 4.2.1 基本原理

Decode 阶段，之前时间步的 K、V 不需要重新计算，可以缓存复用：

$$\text{Attention}(q_t, [K_{1:t}], [V_{1:t}]) = \text{softmax}\left(\frac{q_t K_{1:t}^T}{\sqrt{d_k}}\right) V_{1:t}$$

KV Cache 的显存占用公式：

$$\text{KV Cache Size} = 2 \times n_{layers} \times n_{kv\_heads} \times d_{head} \times \text{seq\_len} \times \text{bytes\_per\_element}$$

### 4.2.2 各模型 KV Cache 占用实测

| 模型 | 层数 | KV Heads | d_head | 8K ctx (FP16) | 128K ctx (FP16) |
|------|------|----------|--------|---------------|-----------------|
| Llama 3 8B | 32 | 8 (GQA) | 128 | 0.5 GB | 8 GB |
| Llama 3 70B | 80 | 8 (GQA) | 128 | ~2.68 GB | 20.5 GB |
| DeepSeek-V3 671B | 61 | MLA (d_c=512) | - | 0.12 GB | 1.9 GB |
| GPT-4 (推测) | 120 | 96 (MHA) | 128 | 23.6 GB | 377 GB |

DeepSeek-V3 的 MLA 压缩效果极其显著——同等上下文长度下，KV Cache 只有 Llama 3 70B 的 1/10。

### 4.2.3 PagedAttention：OS 虚拟内存的借鉴

vLLM (SOSP 2023) 的核心创新 PagedAttention 借鉴了操作系统的虚拟内存分页：

```
传统方式（预分配连续内存）：          PagedAttention（按需分页分配）：
┌──────────────────────────┐      ┌───┬───┬───┬───┬───┬───┐
│ Request 1 KV  [████░░░░] │      │ 1 │ 3 │ 7 │ 2 │ 5 │ 9 │ 物理块
│ Request 2 KV  [██░░░░░░] │      └─┬─┴─┬─┴─┬─┴─┬─┴─┬─┴─┬─┘
│ 碎片浪费        ^^^^^^^^  │        │   │   │   │   │   │
└──────────────────────────┘      ┌─┴───┴───┴───┴───┴───┴─┐
平均浪费 40%+ 显存                  │ 页表：逻辑块 → 物理块   │
                                   │ Req1: [0→1, 1→3, 2→7] │
                                   │ Req2: [0→2, 1→5]      │
                                   └────────────────────────┘
                                   浪费 < 4%（仅最后一个块内部碎片）
```

```python
# PagedAttention 简化实现
class PagedKVCache:
    def __init__(self, block_size=16, num_blocks=1024):
        self.block_size = block_size
        self.free_blocks = list(range(num_blocks))
        self.block_table = {}  # {request_id: [physical_block_ids]}
        # 物理块存储: [num_blocks, 2, num_heads, block_size, head_dim]
        self.kv_blocks = torch.zeros(num_blocks, 2, num_heads, block_size, head_dim)
    
    def allocate(self, request_id):
        """按需分配一个物理块"""
        if not self.free_blocks:
            raise MemoryError("No free blocks!")
        block_id = self.free_blocks.pop()
        if request_id not in self.block_table:
            self.block_table[request_id] = []
        self.block_table[request_id].append(block_id)
        return block_id
    
    def append_token(self, request_id, layer_id, k, v):
        """追加一个 token 的 KV，必要时分配新块"""
        blocks = self.block_table[request_id]
        last_block = blocks[-1]
        slot = self.get_slot_count(request_id) % self.block_size
        if slot == 0 and self.get_slot_count(request_id) > 0:
            last_block = self.allocate(request_id)
        self.kv_blocks[last_block, 0, :, slot, :] = k  # Key
        self.kv_blocks[last_block, 1, :, slot, :] = v  # Value
```

**Copy-on-Write 优化**：相同前缀（如 system prompt）的多个请求可以共享 KV Cache 块，只在内容分叉时才复制，进一步节省显存。

### 4.2.4 MLA 压缩：DeepSeek 的 32× 压缩

DeepSeek-V2/V3 的 Multi-head Latent Attention (MLA) 将 KV 压缩到低维潜在向量：

$$c_t^{KV} = W^{DKV} h_t, \quad c_t^{KV} \in \mathbb{R}^{d_c}, \quad d_c = 512$$

推理时从低维向量重建 K、V：

$$K_t = W^{UK} c_t^{KV}, \quad V_t = W^{UV} c_t^{KV}$$

其中 $d_c = 512 \ll n_h \cdot d_h = 16384$，实现 **32× 压缩**。

**MLA 的数学等价性**：可以证明 MLA 等价于在低秩空间做注意力计算，不损失表达能力，但大幅减少缓存量。关键 trick 是将 $W^{UK}$ 吸收到 $W^Q$ 中，避免运行时的额外矩阵乘法。

### 4.2.5 GQA vs MQA vs MLA 对比

```
MHA (原始 Transformer)     GQA (Llama 3)           MLA (DeepSeek)
Q: [h1 h2 h3 h4 h5 h6]   Q: [h1 h2|h3 h4|h5 h6]  Q: [h1 h2 h3 h4 h5 h6]
K: [h1 h2 h3 h4 h5 h6]   K: [g1   |g2   |g3   ]   K: [c → 低维潜在空间]
V: [h1 h2 h3 h4 h5 h6]   V: [g1   |g2   |g3   ]   V: [c → 低维潜在空间]

KV Cache: 100%             KV Cache: 50%            KV Cache: 3.1%
精度: 基线                  精度: ≈基线              精度: ≈基线
```

## 4.3 量化：用精度换效率

### 4.3.1 量化原理

对称量化的核心映射：

$$w_q = \text{round}\left(\frac{w}{\Delta}\right), \quad \Delta = \frac{\max(|W|)}{2^{b-1} - 1}$$

反量化恢复：

$$\hat{w} = \Delta \cdot w_q \approx w$$

非对称量化额外引入零点 $z$：

$$w_q = \text{round}\left(\frac{w - z}{s}\right), \quad \hat{w} = s \cdot w_q + z$$

### 4.3.2 常见量化方案全景

| 方法 | 精度 | 类型 | 模型大小 | 精度损失 | 速度提升 | 适用场景 |
|------|------|------|---------|---------|---------|---------|
| FP16/BF16 | 16-bit | 权重+激活 | 1× | 无 | 1× | 训练/基准推理 |
| INT8 (W8A8) | 8-bit | 权重+激活 | 0.5× | 极小 | ~1.5× | 通用推理 |
| SmoothQuant | W8A8 | 权重+激活 | 0.5× | 极小 | ~1.7× | A100+ 推理 |
| INT4 (GPTQ) | 4-bit | 仅权重 | 0.25× | 小 | ~2× | 显存受限场景 |
| INT4 (AWQ) | 4-bit | 仅权重 | 0.25× | 小 | ~2.2× | 高吞吐推理 |
| GGUF Q4_K_M | 4-bit 混合 | 仅权重 | 0.28× | 小 | ~2× | CPU/边缘设备 |
| FP8 (E4M3) | 8-bit | 权重+激活 | 0.5× | 极小 | ~1.8× | H100/B200 |
| INT2 (QuIP#) | 2-bit | 仅权重 | 0.125× | 明显 | ~3× | 极端压缩 |

### 4.3.3 GPTQ：逐列量化与误差补偿

GPTQ 基于 OBQ（Optimal Brain Quantization），核心思想是逐列量化并用 Hessian 信息补偿误差：

$$\delta_F = \frac{(\text{quant}(w_q) - w_q)^2}{[H^{-1}]_{qq}}$$

量化第 $q$ 列后，用 Hessian 逆矩阵更新剩余列的权重以补偿量化误差。

```python
# GPTQ 核心算法（简化）
def gptq_quantize(W, H_inv, bits=4, group_size=128):
    """
    W: [out_features, in_features] 权重矩阵
    H_inv: 校准数据计算的 Hessian 逆
    """
    Q = W.clone()
    for col in range(W.shape[1]):
        # 1. 量化当前列
        q = quantize_column(W[:, col], bits)
        # 2. 计算量化误差
        error = (W[:, col] - q) / H_inv[col, col]
        # 3. 补偿后续列
        Q[:, col] = q
        W[:, col+1:] -= error.unsqueeze(1) * H_inv[col, col+1:]
    return Q
```

### 4.3.4 AWQ：激活感知量化

AWQ (Activation-aware Weight Quantization) 的洞察是：**不是所有权重同等重要——与大激活值对应的权重通道更关键**。

$$\hat{w}_c = s_c \cdot \text{round}\left(\frac{w_c}{s_c \cdot \Delta}\right) \cdot \Delta$$

其中缩放因子 $s_c$ 根据激活值大小自适应调整，保护重要通道免受量化损伤。

### 4.3.5 SmoothQuant：平滑激活异常值

LLM 的激活值中存在少数极大的异常值（outliers），直接量化激活会导致精度灾难。SmoothQuant 的解决方案是将激活的"难度"迁移到权重上：

$$Y = (X \text{diag}(s)^{-1}) \cdot (\text{diag}(s) W) = \hat{X} \hat{W}$$

```python
# SmoothQuant 平滑操作
def smooth_activation(X, W, alpha=0.5):
    """将激活异常值平滑到权重"""
    act_scales = X.abs().max(dim=0).values  # 每通道激活最大值
    weight_scales = W.abs().max(dim=0).values  # 每通道权重最大值
    s = (act_scales.pow(alpha) / weight_scales.pow(1-alpha)).clamp(min=1e-5)
    X_smooth = X / s.unsqueeze(0)
    W_smooth = W * s.unsqueeze(1)
    return X_smooth, W_smooth
```

### 4.3.6 FP8 量化：H100/B200 原生支持

NVIDIA H100 及后续 GPU 原生支持 FP8 (E4M3/E5M2) 格式，无需额外量化步骤：

- **E4M3**（4位指数+3位尾数）：更高精度，适合前向推理
- **E5M2**（5位指数+2位尾数）：更大范围，适合反向传播

DeepSeek-V3 在训练中使用了 FP8 前向计算，节省了大量训练成本。

## 4.4 投机采样（Speculative Decoding）

### 4.4.1 核心思想

用一个**小模型（draft model）快速猜测多个 token**，再用大模型**一次性验证**：

```
传统自回归解码（每步 1 token，T_big 延迟/步）：
  大模型: [t1] → [t2] → [t3] → [t4]   = 4 步 × T_big

投机采样（每步猜测 γ=4 个，验证 1 步）：
  小模型: [t1, t2, t3, t4] → 猜 4 个    = 4 步 × T_small (≈忽略)
  大模型: [验证全部]        → 接受 3 个   = 1 步 × T_big
  总计: ≈1 步 × T_big 生成 3 个 token → 3× 加速
```

### 4.4.2 数学保证：无损加速

投机采样的关键数学性质：**输出分布与原始大模型完全一致**（不是近似）。

验证算法使用拒绝采样：

$$p(\text{accept token } x_i) = \min\left(1, \frac{p_{\text{big}}(x_i | x_{<i})}{p_{\text{small}}(x_i | x_{<i})}\right)$$

如果 draft token 被拒绝，从修正分布中重采样：

$$p_{\text{resample}}(x) = \text{normalize}\left(\max\left(0, p_{\text{big}}(x) - p_{\text{small}}(x)\right)\right)$$

### 4.4.3 加速比分析

$$\text{Speedup} = \frac{1 - \alpha^{\gamma+1}}{(1 - \alpha)(c \gamma + 1)}$$

其中：$\gamma$ = 猜测长度，$\alpha$ = 平均接受率，$c$ = $T_{\text{small}} / T_{\text{big}}$

实际效果：

| Draft/Target 配对 | 接受率 α | γ | 加速比 | 场景 |
|-------------------|---------|---|--------|------|
| Llama 3 8B → 70B | 0.75 | 5 | 2.3× | 通用 |
| TinyLlama → 70B | 0.55 | 4 | 1.6× | 通用 |
| Medusa heads → 70B | 0.80 | 3 | 2.5× | 无需额外模型 |
| EAGLE-2 → 70B | 0.82 | 5 | 3.1× | SOTA |

### 4.4.4 Medusa：多头并行猜测

Medusa 给大模型加几个轻量级的预测头，每个头预测未来不同位置的 token：

```python
class MedusaHead(nn.Module):
    """Medusa 辅助预测头"""
    def __init__(self, hidden_size, vocab_size, num_medusa_heads=4):
        super().__init__()
        self.heads = nn.ModuleList([
            nn.Sequential(
                nn.Linear(hidden_size, hidden_size),
                nn.SiLU(),
                nn.Linear(hidden_size, vocab_size),
            ) for _ in range(num_medusa_heads)
        ])
    
    def forward(self, hidden_states):
        """每个头预测未来第 i+1 个位置的 token"""
        return [head(hidden_states) for head in self.heads]
```

优势：不需要额外的 draft 模型，只需训练几个轻量级的线性头（参数量 < 1% 的主模型），即可实现 2-3× 加速。

### 4.4.5 EAGLE-2：特征级投机采样

EAGLE-2 (2024) 不是在 token 级别猜测，而是在隐层特征级别进行投机：

1. 用轻量级网络预测下一步的隐层表示
2. 构建 token 树结构（而非线性链），一次验证多个分支
3. 通过树注意力机制并行验证所有候选

EAGLE-2 在 MT-Bench 上实现了 3.1× 的无损加速，是当前 SOTA。

## 4.6 前沿：Disaggregated Serving（PD 分离）

### 4.6.1 为什么要分离 Prefill 和 Decode？

传统做法将 Prefill 和 Decode 放在同一组 GPU 上交替执行。问题在于：

- Prefill 是计算密集型，需要高算力
- Decode 是访存密集型，需要高带宽
- 二者混合执行时，**Prefill 会抢占 Decode 的延迟 SLO**

DistServe (OSDI 2024) 的解决方案：

```
                    ┌──────────────────┐
用户请求 ──────→    │  Prefill 集群     │ ─── KV Cache ──→  ┌──────────────┐
                    │  (计算密集型 GPU)  │     传输          │  Decode 集群  │ ──→ 输出
                    │  H100 SXM         │                   │ (带宽优化 GPU) │
                    └──────────────────┘                   └──────────────┘
```

### 4.6.2 NVIDIA Dynamo：生产级 PD 分离

NVIDIA Dynamo (2026) 是首个生产级的 PD 分离推理框架：

- **自动路由器**：根据输入长度、当前负载自动分配 Prefill/Decode 实例
- **KV Cache 传输优化**：NVLink/RDMA 直传，无需落盘
- **弹性扩缩容**：Prefill 和 Decode 集群可以独立扩缩

实测效果：相比 vLLM 统一部署，Dynamo PD 分离在保持相同 P99 延迟下，**吞吐提升 2.5×**。
