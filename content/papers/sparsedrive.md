# SparseDrive: End-to-End Autonomous Driving via Sparse Scene Representation

> **论文**: SparseDrive: End-to-End Autonomous Driving via Sparse Scene Representation  
> **作者**: Wencheng Han, Xiaokang Chen, Zhengzhu Ye, Jianbing Shen  
> **机构**: Beijing Institute of Technology / INRIA  
> **发表**: arXiv 2024 / ECCV 2024  
> **领域**: 自动驾驶 · 端到端规划 · 稀疏感知  
> **重要度**: ⭐⭐⭐⭐⭐ (5/5) — 首个实现 45 FPS 实时端到端驾驶的方案  
> **arXiv**: https://arxiv.org/abs/2405.19620  
> **GitHub**: https://github.com/swc-17/SparseDrive

## 一句话总结

SparseDrive 用**稀疏 3D 实例表示**替代密集 BEV，并将感知、预测、规划设计为**并行解码**而非串行，在 nuScenes 上超越 UniAD 和 VAD 的同时，实现 **45 FPS** 实时推理——比 UniAD 快 **25x**。

## 核心问题：串行架构的效率瓶颈

### UniAD/VAD 的串行问题

```
UniAD/VAD 的串行架构:

  感知 → 预测 → 规划
  (必须等感知完成才能做预测，等预测完成才能做规划)
  
  问题:
  1. 延迟累积: 每个模块的延迟叠加
     感知 200ms + 预测 150ms + 规划 100ms = 450ms
  
  2. 误差传播: 感知错误 → 预测错误 → 规划错误
     无法在规划阶段"纠正"感知的错误
  
  3. 梯度流受限: 规划梯度需要穿过多个模块才能到达感知
     梯度消失问题更严重
  
  4. 无法实时: 450ms 延迟 → 2.2 FPS，远低于实时要求
```

### SparseDrive 的核心洞察

> **感知、预测、规划可以并行！** 它们共享同一个稀疏场景表示，可以同时解码。

```
SparseDrive 的并行架构:

  多相机图像
       ↓
  稀疏 3D 实例特征提取
  (每个实例 = 一个 3D 高斯)
       ↓
  ┌────────────────────────────────────┐
  │         并行解码 (同时进行)          │
  │                                    │
  │  ┌──────────┐  ┌──────────────┐   │
  │  │ 感知头    │  │  运动预测头   │   │
  │  │ (检测)   │  │  (轨迹预测)  │   │
  │  └──────────┘  └──────────────┘   │
  │                                    │
  │  ┌──────────────────────────────┐  │
  │  │         规划头               │  │
  │  │  (直接从稀疏特征规划)         │  │
  │  └──────────────────────────────┘  │
  └────────────────────────────────────┘
       ↓
  感知结果 + 预测结果 + 规划轨迹
  
  延迟: 感知/预测/规划 并行 → 总延迟 ≈ 单个模块延迟
  FPS: 45 FPS (vs UniAD 1.8 FPS)
```

## 架构设计

### 稀疏 3D 实例表示

```python
# SparseDrive 的核心数据结构: 稀疏 3D 实例
class SparseInstance:
    # 3D 位置 (在 BEV 空间)
    center: Tensor      # [3] (x, y, z)
    
    # 3D 尺寸
    size: Tensor        # [3] (l, w, h)
    
    # 朝向
    yaw: float          # 偏航角
    
    # 语义特征
    feat: Tensor        # [D=256] 从相机图像提取的特征
    
    # 速度 (用于时序跟踪)
    velocity: Tensor    # [2] (vx, vy)

# 场景表示: N 个稀疏实例
# N ≈ 300 (vs UniAD 的 200×200=40000 个 BEV 格子)
# 稀疏度: 300 vs 40000 → 133x 更稀疏
```

### 稀疏特征提取

```
SparseDrive 的特征提取:

  传统 BEV 方法:
  多相机图像 → 2D 特征 → 密集 BEV 特征 (200×200)
  问题: 需要将所有像素特征聚合到 BEV，计算量大
  
  SparseDrive:
  多相机图像 → 2D 特征 → 稀疏 3D 实例特征
  
  具体步骤:
  1. 初始化稀疏实例 Query (可学习的 3D 位置 + 特征)
  
  2. Sparse Cross-Attention:
     每个实例 Query 只关注对应的相机区域
     (根据 3D 位置投影到各相机，只采样对应像素)
     
  3. 实例特征更新:
     Query ← 采样到的相机特征
     → 每个实例获得丰富的视觉特征
  
  4. 实例间交互:
     Sparse Self-Attention
     实例之间相互关注，建模交互关系
```

### 并行解码设计

```
SparseDrive 的并行解码:

  输入: N 个稀疏实例特征 {f_1, ..., f_N}
  
  并行解码 1: 感知头
  ┌─────────────────────────────────────┐
  │ 对每个实例 f_i:                      │
  │   类别分类: MLP(f_i) → 类别概率      │
  │   3D 框回归: MLP(f_i) → (x,y,z,l,w,h,θ) │
  │   置信度: MLP(f_i) → score          │
  └─────────────────────────────────────┘
  
  并行解码 2: 运动预测头
  ┌─────────────────────────────────────┐
  │ 对每个实例 f_i:                      │
  │   多模态轨迹: MLP(f_i) → K×T×2      │
  │   (K=6 条轨迹，T=6 个时间步)         │
  │   轨迹置信度: MLP(f_i) → K 个分数   │
  └─────────────────────────────────────┘
  
  并行解码 3: 规划头
  ┌─────────────────────────────────────┐
  │ Ego Query (自车特征)                 │
  │   + 所有实例特征 (通过 Attention)    │
  │   → GRU 规划器                      │
  │   → 未来 3s 规划轨迹                 │
  └─────────────────────────────────────┘
  
  关键: 三个解码头同时运行，无需等待
```

### 时序建模

```
SparseDrive 的时序处理:

  传统方法 (BEVFormer TSA):
  历史 BEV 特征 → 当前 BEV 特征
  问题: 需要存储大量历史 BEV 特征 (200×200×256)
  
  SparseDrive:
  历史稀疏实例 → 当前稀疏实例
  
  具体:
  1. 历史实例跟踪:
     t-1 时刻的实例 → 根据速度预测 t 时刻位置
     → 与 t 时刻新检测的实例匹配
  
  2. 特征传递:
     匹配成功的实例: 历史特征 + 当前特征 → 融合
     新出现的实例: 仅使用当前特征
  
  优势:
  - 存储量: 300 个实例 × 256 维 vs 200×200×256 BEV
  - 存储减少: 133x
  - 时序信息更精准: 实例级别的跟踪 vs 栅格级别的对齐
```

## 核心创新点

### 1. 并行解码架构

SparseDrive 打破了感知→预测→规划的串行范式：
- 三个任务共享同一个稀疏特征表示
- 并行解码消除了串行延迟
- 规划头可以直接访问原始实例特征，不受感知误差影响

### 2. 稀疏表示的优势

```
为什么稀疏表示比密集 BEV 更好？

1. 精度更高:
   密集 BEV: 每个格子平均特征，信息被稀释
   稀疏实例: 每个实例有专属特征，信息更集中
   
2. 速度更快:
   密集 BEV: 处理 40,000 个格子
   稀疏实例: 处理 300 个实例
   
3. 泛化更好:
   密集 BEV: 依赖固定分辨率（200×200）
   稀疏实例: 自适应实例数量，不依赖固定网格
```


## 局限性

```
SparseDrive 的局限:
1. 稀疏实例数量固定 (N=300)
   → 密集场景（停车场/十字路口）可能不够
   
2. 仅在 nuScenes 验证
   → 未在真实闭环场景测试
   
3. 开环评估的局限
   → L2 误差不能完全反映真实驾驶性能
   
4. 缺乏语言理解能力
   → 无法处理导航指令、交通标志语义
   → 后续 OpenDriveVLA 解决了这个问题
```
