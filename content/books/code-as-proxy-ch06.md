## 6.1 为什么这四个行业？

Code as Proxy 的价值在数据越敏感的行业越显著。我们选择四个代表性行业：

| 行业 | 数据敏感度 | 监管强度 | AI 需求 | Code as Proxy 价值 |
|------|-----------|---------|---------|-------------------|
| **金融** | 🔴 极高 | 🔴 极强 | 🔴 极高 | ⭐⭐⭐⭐⭐ |
| **医疗** | 🔴 极高 | 🔴 极强 | 🟡 高 | ⭐⭐⭐⭐⭐ |
| **国防** | 🔴 极高 | 🔴 极强 | 🔴 极高 | ⭐⭐⭐⭐⭐ |

## 6.2 金融行业：交易数据不出域

### 6.2.1 场景描述

一家大型银行希望使用 AI 分析交易数据，检测异常交易和洗钱模式。但交易数据受到多重监管约束：

- **PCI-DSS**：支付卡数据必须加密存储和传输
- **银保监会**：客户金融数据不得传输到境外
- **反洗钱法**：交易监控数据属于高度机密
- **GDPR**（欧洲业务）：客户 PII 不得未经授权处理

### 6.2.2 Code as Proxy 架构

```
┌─────────────────────────────────────────────────┐
│                 银行内部网络                       │
│                                                   │
│  ┌──────────────┐    ┌──────────────────────┐   │
│  │ 交易数据库    │    │ Schema Registry       │   │
│  │ (PCI-DSS     │    │ ┌──────────────────┐ │   │
│  │  加密存储)    │    │ │ 交易 Schema      │ │   │
│  └──────┬───────┘    │ │ - 交易ID (非PII) │ │   │
│         │            │ │ - 金额范围       │ │   │
│         │            │ │ - 交易类型       │ │   │
│         │            │ │ - 时间戳格式     │ │   │
│         │            │ │ - 风险等级定义   │ │   │
│         │            │ └──────────────────┘ │   │
│         │            └──────────┬───────────┘   │
│         │                       │                │
│         │            ┌──────────▼───────────┐   │
│         │            │ AI 安全网关           │   │
│         │            │ (Schema → 外部 LLM)   │   │
│         │            └──────────┬───────────┘   │
│         │                       │                │
│         │            ┌──────────▼───────────┐   │
│         │            │ 代码生成 (外部 LLM)   │   │

└─────────────────────────────────────────────────┘
```

### 6.2.3 金融 Ontology 示例

```yaml
entities:
  Transaction:
    fields:
      - name: transaction_id
        type: string
        pii: false
        description: "交易唯一标识"
      - name: amount
        type: decimal
        pii: false
        description: "交易金额（人民币）"
        statistics: { min: 0.01, max: 50000000, p50: 2800, p99: 500000 }
      - name: transaction_type
        type: enum
        values: [transfer, payment, withdrawal, deposit, forex]
        description: "交易类型"
      - name: risk_score
        type: float
        description: "风险评分 0-1，>0.8 为高风险"
      - name: customer_id
        type: string
        pii: true  # ← PII 标记，AI 不可见
        description: "[PII] 客户标识"
    
    actions:
      - name: detect_anomalies
        description: "检测异常交易模式"
        output: "异常交易统计摘要（不含客户信息）"
        requires_approval: false
      
      - name: flag_suspicious
        description: "标记可疑交易，触发人工审查"
        requires_approval: true
        approval_role: "compliance_officer"
    
    constraints:
      - "单笔交易 > ¥50,000 自动触发大额报告"
      - "同一客户 24h 内累计 > ¥200,000 触发频繁交易预警"
      - "跨境交易需要额外合规检查"
```

## 6.3 医疗行业：患者数据零接触

### 6.3.1 场景描述

一家三甲医院希望使用 AI 分析电子病历（EMR）数据，辅助临床决策和医学研究。但医疗数据受到最严格的隐私保护：

- **HIPAA**（美国）：PHI（受保护健康信息）的使用有严格限制
- **《个人信息保护法》**（中国）：医疗数据属于敏感个人信息
- **医疗数据安全管理办法**：医疗数据不得未经授权出院

### 6.3.2 医疗 Code as Proxy 架构

```
场景：AI 辅助分析糖尿病患者的血糖控制效果

传统方式（❌ 不合规）：
  患者病历数据 → 外部 AI → "患者张三的 HbA1c 从 8.2 降到 6.8"
  问题：患者姓名、病历号、具体检验值全部暴露

Code as Proxy 方式（✅ 合规）：
  EMR Schema → AI → 生成分析代码 → 院内执行 → 摘要
  AI 看到：
    "EMR 包含字段：patient_id(PII), diagnosis_code(ICD-10), 
     lab_result(type, value, unit, date), medication(name, dose)"
  AI 生成：
    "筛选 diagnosis_code = E11（2型糖尿病）的患者，
     计算 HbA1c 的 6 个月变化趋势，按用药方案分组统计"
  AI 收到的摘要：
    "共 1,847 名患者，二甲双胍组平均 HbA1c 下降 1.2%，
     GLP-1 组下降 1.8%，差异有统计学意义 (p<0.01)"
```

### 6.3.3 医学研究中的应用

Code as Proxy 特别适合**多中心临床研究**——多家医院的数据不出院，AI 生成统一的分析代码分发到各医院执行：

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 医院 A    │  │ 医院 B    │  │ 医院 C    │
│ 患者数据  │  │ 患者数据  │  │ 患者数据  │
│ (不出院)  │  │ (不出院)  │  │ (不出院)  │
└────┬─────┘  └────┬─────┘  └────┬─────┘
     │              │              │
     │   统一 Schema │              │
     │   ┌──────────▼──────────┐  │
     │   │    AI 生成分析代码    │  │
     │   │  (只看到统一 Schema)  │  │
     │   └──────────┬──────────┘  │
     │              │              │
     ▼              ▼              ▼
  本地执行       本地执行       本地执行
     │              │              │
     ▼              ▼              ▼
  摘要 A         摘要 B         摘要 C
     │              │              │
     └──────────────┼──────────────┘
                    ▼
              联合摘要聚合
         "3 家医院共 5,231 名患者，
          总体 HbA1c 下降 1.5%..."
```

这就是**联邦代码执行（Federated Code Execution）**——联邦学习的 Code as Proxy 版本。

## 6.4 国防行业：最高机密环境中的 AI

### 6.4.1 场景描述

军事情报分析需要处理多源情报数据（信号情报 SIGINT、人力情报 HUMINT、开源情报 OSINT、地理空间情报 GEOINT），这些数据的分级从 CONFIDENTIAL 到 TOP SECRET/SCI。

### 6.4.2 Palantir Gotham + TITAN 的实践

### 6.4.3 国防场景的特殊要求

| 要求 | Code as Proxy 的满足方式 |
|------|------------------------|
| 完全断网运行 | TITAN 本地部署，无外部依赖 |
| 多级安全（MLS） | Ontology 的 ACL 支持多级安全标签 |
| 审计追溯 | 每个 Action 完整记录，不可篡改 |
| 人工审批 | 高风险 Action 必须经过指挥官确认 |
| 抗干扰 | 本地 LLM 不受外部 Prompt Injection 影响 |

## 6.5 自动驾驶行业：传感器数据的安全分析

### 6.5.1 场景描述

自动驾驶公司拥有 PB 级的传感器数据（摄像头、LiDAR、雷达、GPS），需要 AI 辅助进行数据分析、场景挖掘和模型评估。但这些数据包含：

- **道路场景数据**：可能包含行人面部、车牌等 PII
- **驾驶行为数据**：可能推断驾驶员身份和习惯
- **高精地图数据**：可能涉及国家安全（中国测绘法）
- **核心算法数据**：训练数据和标注是核心 IP

### 6.5.2 自动驾驶 Code as Proxy 架构

```
场景：AI 辅助分析 Corner Case 场景分布

传统方式（❌ 风险高）：
  将传感器数据上传到云端 AI → AI 直接分析图像和点云
  问题：PII 泄露、高精地图数据出境、核心数据暴露

Code as Proxy 方式（✅ 安全）：
  数据 Schema → AI → 生成分析代码 → 本地集群执行 → 摘要

AI 看到的 Schema：
  {
    "dataset": "driving_scenarios",
    "fields": [
      {"name": "scenario_id", "type": "string"},
      {"name": "scenario_type", "type": "enum", 
       "values": ["highway", "urban", "intersection", "parking"]},
      {"name": "weather", "type": "enum",
       "values": ["clear", "rain", "snow", "fog", "night"]},
      {"name": "num_vehicles", "type": "int", "stats": {"mean": 8, "max": 50}},
      {"name": "num_pedestrians", "type": "int", "stats": {"mean": 3, "max": 30}},
      {"name": "has_corner_case", "type": "bool", "distribution": {"true": "12%"}},
      {"name": "corner_case_type", "type": "enum",
       "values": ["cut_in", "jaywalking", "construction", "emergency_vehicle", ...]}
    ]
  }

AI 生成的分析代码：
  "按场景类型和天气条件交叉统计 Corner Case 分布，
   计算各类型的出现频率和严重程度分布"

AI 收到的摘要：
  "共 2.3M 场景，Corner Case 占比 12.3%。
   城市交叉路口 + 雨天的 Corner Case 密度最高（23.7%），
   主要类型：行人横穿(34%)、车辆加塞(28%)、施工区域(18%)"
```

### 6.5.3 数据闭环中的 Code as Proxy

在自动驾驶的数据闭环中，Code as Proxy 可以应用于多个环节：

| 环节 | 传统方式 | Code as Proxy 方式 |
|------|---------|-------------------|
| **场景挖掘** | AI 直接分析视频/点云 | AI 根据元数据 Schema 生成筛选代码 |
| **数据质量检测** | AI 检查标注质量 | AI 生成质量检测规则代码 |
| **模型评估** | AI 分析预测结果 | AI 生成评估指标计算代码 |
| **数据统计** | AI 直接查询数据库 | AI 根据 Schema 生成统计查询 |

注意：自动驾驶中的**感知模型训练**不适用 Code as Proxy（模型必须"看到"图像），但**数据管理和分析**环节非常适用。

## 6.6 跨行业通用模式

### 6.6.1 实施路径

无论哪个行业，Code as Proxy 的实施都遵循类似的路径：

```
Phase 1：Schema 建设（1-2 月）
  ├── 梳理核心数据资产
  ├── 构建增强 Schema（含业务语义）
  ├── 标记 PII 和敏感字段
  └── 建立 Schema Registry

Phase 2：原型验证（1-2 月）
  ├── 选择 1-2 个低风险场景
  ├── 搭建沙箱执行环境
  ├── 验证代码生成质量
  └── 收集用户反馈

Phase 3：安全加固（1-2 月）
  ├── 部署代码安全验证器
  ├── 实现结果摘要策略
  ├── 建立审计日志系统
  └── 通过安全评审

Phase 4：规模化推广（持续）
  ├── 扩展到更多数据源和场景
  ├── 优化 Schema 质量
  ├── 建立 Ontology 模板库
  └── 培训用户和运维团队
```

## 本章小结

| 行业 | 核心挑战 | Code as Proxy 价值 | 特殊要求 |
|------|---------|-------------------|---------|
| **金融** | PCI-DSS + 反洗钱 | 交易数据不出域，合规审计简化 | 差分隐私 + 实时性 |
| **医疗** | HIPAA + 患者隐私 | 病历数据零接触，多中心研究 | 联邦代码执行 |
| **国防** | IL6 最高机密 | 完全断网运行，情报不泄露 | TITAN 边缘部署 |
| **自动驾驶** | PII + 高精地图 | 传感器数据本地分析 | 大规模数据处理 |

> **核心洞察**：Code as Proxy 不是一种通用方案，而是一种**按行业定制的架构模式**。每个行业有不同的数据敏感度、监管要求和 AI 需求，Code as Proxy 的实现需要根据行业特点进行调整。但核心原则不变：**AI 写代码，数据不出域。**
